Section 1 — Project Overview

What does this application do and what powers it?

1What the Project Does

The AI-Powered Resume Analyzer is a full-stack SaaS web application that helps job seekers optimize their resumes against specific job descriptions using artificial intelligence. Users can:

  • Register and log in securely.
  • Upload their resume in PDF or DOCX format.
  • Optionally paste a target job description (or use a smart default).
  • Receive a real-time ATS match score, strengths, weaknesses, and recommendations generated by Google Gemini.
  • View resume history, job match reports, AI skill suggestions, interview prep questions, and account settings — all from a unified dashboard.
Business Value
This solves a real, high-demand problem: over 75% of resumes are rejected by Applicant Tracking Systems before a human reads them. This tool gives candidates an unfair advantage by exposing exactly what ATS systems look for.

Section 3 — Backend Deep-Dive

Every backend file explained step by step for an interviewer.

1Server Entry Point — server/index.js

This is the root of the Express application. It loads environment variables via dotenv, configures CORS to only accept requests from the Next.js app at localhost:3000, parses incoming JSON bodies, registers all route handlers, attaches global error middleware, and starts listening on port 3001.

Key Design Decision — Shared Prisma Singleton
The PrismaClient is instantiated once in index.js and exported as a named export (export const prisma). All controllers import this single instance. This prevents connection pool exhaustion — a common mistake is creating a new PrismaClient() per request.

2Authentication — middleware/auth.js & controllers/authController.js

  • Registration — POST /api/auth/register

    Accepts { name, email, password }. Checks if email already exists using Prisma. Hashes the password using bcrypt with a salt factor of 10 (industry standard). Creates the User and a default UserSettings record in a single atomic Prisma transaction. Returns the user object + a signed JWT token (30-day expiry).

  • Login — POST /api/auth/login

    Finds the user by email, then uses bcrypt.compare() to verify the submitted password against the stored hash. If valid, returns a new JWT. If invalid, returns 401 Unauthorized.

  • JWT Middleware — protect()

    Reads the Authorization: Bearer <token> header from every protected request. Verifies the token using the JWT_SECRET from .env. Fetches the user from the database and attaches them to req.user for downstream controllers to use. Returns 401 if the token is missing or expired.

3File Parsing — services/fileParser.js

When a resume file arrives (buffered in memory by Multer), extractTextFromFile(buffer, mimetype) is called:

  • PDF: uses pdf-parse library to extract raw text from the PDF binary buffer.
  • DOCX / DOC: uses mammoth library to extract raw text from Word document XML.
  • After extraction, a regex (/\s+/g) normalizes excessive whitespace to a single space — this is critical because AI token limits are expensive, and clean text improves AI accuracy.

4AI Analysis — services/aiService.js

This is the intelligence core. The analyzeResume(resumeText, jobDescription) function:

  • Sends the prompt containing the resume + job description to Google Gemini 2.5 Flash.
  • Forces structured JSON output using response_format: { type: 'json_object' } — ensuring the AI always returns parseable JSON.
  • Uses a low temperature (0.2) to get consistent, factual analysis rather than creative outputs.
  • The response is parsed and returns: matchScore, strengths[], weaknesses[], recommendations[], summary.
Interview Tip — Why temperature=0.2?
Temperature controls "creativity" in AI responses. 0 = deterministic/robotic, 1 = highly creative. For factual analysis tasks like ATS scoring, we want consistent, repeatable results — hence 0.2.

5ORM & Database — Prisma + PostgreSQL

Prisma provides a type-safe query builder that compiles our schema.prisma model definitions into SQL. Key decisions:

  • @id @default(uuid()) — uses UUIDs instead of sequential integers (prevents ID enumeration attacks).
  • Json? fields for strengths, weaknesses, etc. — stores arrays/objects natively in PostgreSQL's jsonb column type.
  • onDelete: Cascade — cascading deletes ensure no orphaned records when a user is deleted.
  • @db.Text on long text fields — prevents PostgreSQL VARCHAR length limits on job descriptions and summaries.

Section 6 — Bug Audit & Fixes Applied

All bugs found during the full codebase analysis and the exact fixes applied.

1Critical Bugs Found & Fixed

Bug #1 — CRITICAL: Wrong middleware import in resumeRoutes.js

File: server/routes/resumeRoutes.js

Problem: The route was importing authenticateToken from ../middleware/authMiddleware.js — a file that does not exist. This would cause the server to crash on startup with a module-not-found error.

Fix: Changed the import to use protect from ../middleware/auth.js, consistent with all other routes.

// BEFORE (broken)
import { authenticateToken } from '../middleware/authMiddleware.js';

// AFTER (fixed)
import { protect } from '../middleware/auth.js';
Bug #2 — CRITICAL: Wrong Prisma field names in resumeController.js

File: server/controllers/resumeController.js

Problem: The controller was trying to create a Resume record using fields filename and content, and a JobMatch using matchScore and analysis. None of these fields exist in the Prisma schema — the correct names are name, size, score, filePath, matchPercentage, etc. This would cause a Prisma runtime error on every upload attempt.

Fix: Rewrote the Prisma create calls to match the exact field names in schema.prisma.

Bug #3 — CRITICAL: Duplicate PrismaClient instantiation

File: server/controllers/resumeController.js

Problem: The controller was creating a new PrismaClient() instance at the top of the file, in addition to the one exported from index.js. In a production environment, this causes connection pool exhaustion as each import creates a new database connection pool.

Fix: Removed the local PrismaClient instantiation and imported the shared singleton from ../index.js.

Bug #4 — Wrong API response mapping in useResumeStore.tsx

File: client/src/hooks/useResumeStore.tsx

Problem: The frontend was trying to read data.resume.filename and data.analysis.analysis.matchScore, but after fixing the backend the actual response shape is data.resume.name and data.resume.score (since all analysis data is now stored on the Resume record directly).

Fix: Updated the response mapping to read from data.resume using the correct field names matching the Prisma schema.

Bug #5 — Multer file size limit inconsistency

Files: server/middleware/upload.js vs server/routes/resumeRoutes.js

Problem: Two separate Multer configurations exist — upload.js sets a 5MB limit, but resumeRoutes.js defines its own inline Multer with a 10MB limit. The inline one in the route wins, creating an inconsistency with what the frontend validates (5MB).

Recommendation: Use only the shared upload.js middleware in the route file and remove the inline Multer definition.

Section 7 — How to Run the Project

Complete setup guide from zero to running application.

1Prerequisites

  • Node.js v18+ installed
  • PostgreSQL server running locally (default port 5432)
  • A Google Gemini API key (via Google AI Studio)

2Step-by-Step Setup

  • Configure Environment Variables

    Edit server/.env and fill in your PostgreSQL credentials and Gemini API key.

    PORT=3001
CLIENT_URL=http://localhost:3000
DATABASE_URL="postgresql://<user>:<password>@localhost:5432/resumeai"
JWT_SECRET="your-very-long-secret-key"
GEMINI_API_KEY="AIza..."

  • Run Database Migrations

    From the server/ directory, run Prisma migrate to create all tables in PostgreSQL.

    # Inside /server directory
npx prisma migrate dev --name init

  • Start the Backend Server

    # Inside /server directory
npm run dev     # uses nodemon for hot-reload
# Server starts at http://localhost:3001

  • Start the Frontend (Next.js)

    # Inside /client directory
npm install
npm run dev
# App available at http://localhost:3000

  • Test the Full Flow

    Navigate to localhost:3000 → click "Get Started Free" → Register → Go to Dashboard → Upload a PDF resume (optionally paste a job description) → View AI analysis results.

3Health Check

Once the server is running, you can verify it with:

curl http://localhost:3001/api/health
// Expected: { "status": "ok", "message": "AI Resume Analyzer API is running" }

Section 8 — Suggested Additional Features

High-impact features to implement next, ranked by value and complexity.

1High Priority (Core Gaps)

  • GET /api/resumes — Resume History from DB
    Currently, uploaded resumes only persist in React state and are lost on page refresh. Add a backend endpoint to fetch a user's resume list from PostgreSQL on dashboard load.
  • DELETE /api/resumes/:id — Persistent Delete
    The delete button in ResumeHistoryView only removes from local state. Wire it to a real Prisma delete for data integrity.
  • POST /api/resumes/interview — AI Interview Question Generator
    Add a dedicated endpoint that takes the resume text and a category (technical/hr/project) and returns 5 tailored questions from Gemini. Wire it to the InterviewView's "Generate" button.
  • Wire SettingsView to the Backend
    The Profile, Password, and AI model settings panels exist but use local state. Connect them to PUT /api/users/profile and PUT /api/users/settings.

2Medium Priority (Product Differentiators)

  • 🔐 Google OAuth Login
    Integrate NextAuth.js or Passport.js Google strategy. Dramatically reduces sign-up friction — most SaaS tools see 60-70% higher conversion with social login.
  • 📊 Resume Score Trend Chart
    Show a line chart (using Recharts or Chart.js) displaying how the user's ATS score improves over multiple resume versions. Requires storing version history in the database.
  • ✏️ Inline Resume Editor
    Let users edit their resume text directly in the browser, re-run AI analysis, and see score changes in real time without re-uploading a file.
  • 📄 PDF Export of Analysis Report
    Use jsPDF or a server-side PDF library to export the full ATS report (score, keywords, recommendations) as a branded PDF the user can share with mentors or coaches.
  • 🔔 Email Notifications
    When emailAlerts: true is set in UserSettings (backend already stores this), send an email after analysis completes using SendGrid or Nodemailer. The toggle is already in the UI and DB.

3Advanced Features (Scale & Monetization)

  • 🏦 Subscription Tiers + Stripe Integration
    The tier field already exists on the User model. Add a Stripe webhook to upgrade users to "Pro" after payment, gating features like unlimited uploads and GPT-4 access.
  • 🧠 Multi-Model AI Support
    Let users choose between GPT-4o, Claude 3.5 Sonnet, or Gemini 1.5 Pro in Settings. Implement an AI provider factory pattern in aiService.js that routes to the selected model.
  • 🌐 LinkedIn Job Scraping
    Let users paste a LinkedIn job URL instead of copy-pasting the job description. Use a backend scraper (Puppeteer or Cheerio) to extract the JD automatically.
  • 🤝 Recruiter Dashboard
    Add a separate recruiter role that can bulk-upload candidate resumes, run AI comparisons, and rank candidates against a job description. This opens B2B revenue.
